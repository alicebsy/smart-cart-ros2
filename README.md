## 2. 프로젝트 설명 및 목표
* **주제**: 물류 창고 및 마트 환경에서 사용자를 실시간 인지하고 일정 거리를 유지하며 안전하게 자동으로 따라다니는 능동형 사용자 추종 스마트 카트 시스템 구현입니다.
* **프로젝트 목표**:
  1. **Aruco 마커 기반 정밀 추종**: 특정 ID를 가진 Aruco 마커를 카메라로 실시간 인식하여, 대상을 놓치지 않고 1.0m~1.5m의 일정한 거리를 유지하며 추종하게 합니다.
  2. **LiDAR 기반 다중 레이어 안전 시스템**: 비전 인식의 한계나 예기치 못한 돌발 상황에 대비하여, LiDAR 센서가 전방 장애물을 감지하면 추종 명령보다 우선하여 즉각 정지하는 2중 안전 장치를 구축합니다.
  3. **시스템 통합 및 검증**: 인식(Perception), 판단(Decision), 제어(Control) 노드가 유기적으로 메시지를 주고받는 완결된 ROS2 시스템을 Gazebo 시뮬레이션 환경에서 검증합니다.

## 3. 시스템 구조 (System Architecture)

본 시스템은 ROS2 인프라 하에서 다음과 같은 노드 및 토픽 구조로 유기적으로 통신합니다.

### 데이터 흐름 구조
* Gazebo Sim (LiDAR / Camera) -> [/scan (LaserScan) / /camera (Image)] -> cart_control 노드 (알고리즘 연산 및 판단) -> [/cmd_vel 토픽 (Twist msg)] -> Gazebo Sim (터틀봇 바퀴 모터 구동)

1. **Perception (인식)**: Gazebo Sim 가상 세계 내 로봇에 장착된 카메라와 2D LiDAR 센서가 타겟 및 장애물을 감지하여 데이터를 발행합니다.
2. **Decision & Control (판단 및 제어)**: `cart_control` 노드가 센서 데이터를 구독하여 알고리즘 연산을 통해 선속도와 각속도 비례 제어 명령을 생성합니다.
3. **Actuation (구동)**: 생성된 제어 명령이 `/cmd_vel` 토픽을 통해 가상 로봇의 바퀴 모터로 전달되어 최종 추종 및 조향 주행을 수행합니다.

---

## 4. 구체적인 구현 방법

### 4.1 Perception: 비전 및 센서 데이터 처리
* **Aruco 마커 인식**: 연산 효율성이 높은 Aruco 마커(4x4_50)를 활용합니다. OpenCV 라이브러리를 통해 카메라 프레임 내 마커의 4개 모서리 좌표를 검출하고 중심점(x, y)과 화면 내 면적을 계산하여 타겟의 방향과 거리를 추정합니다.
* **LiDAR 필터링**: 전체 360도 스캔 데이터 중 로봇 진행 방향인 전방 영역(약 -20도 ~ +20도)의 거리 데이터만 추출하여 실시간으로 모니터링합니다. 이는 복잡한 지도 생성 없이 로봇 앞의 물체를 가장 빠르게 감지하는 효율적인 방식입니다.

### 4.2 Decision & Control: 제어 알고리즘 및 안전 로직
* **P-Control(비례 제어) 선속도 제어**: 마커의 크기(화면 점유율)를 기준으로 거리를 계산합니다. 마커가 멀어지면 전진 속도를 높이고, 목표 거리 내에 들어오면 감속 및 정지합니다.
* **P-Control(비례 제어) 각속도 제어**: 마커의 중심 좌표가 화면 정중앙에서 벗어난 정도에 비례하여 로봇의 회전 속도를 조절함으로써 대상을 자연스럽게 조향 추종합니다.
* **인터럽트 방식의 긴급 제동 (Safety Layer)**: LiDAR 센서의 전방 측정값이 안전 임계치(0.5m) 이하로 떨어지면, 비전 노드에서 생성된 모든 주행 명령을 강제로 차단(Override)하고 `cmd_vel` 메시지의 모든 값을 0으로 고정하여 안전을 보장합니다.

## 5. AI 사용 여부 및 사용 내용
* **사용 여부**: 본 과제의 시연용 주행 제어 알고리즘 코드 정제 및 최신 가상 시뮬레이션 환경 빌드 에러 디버깅 과정에서 대형 언어 모델(LLM)을 보조 도구로 활용하였습니다.
* **사용 내용**: 구형 Gazebo Classic과 최신 릴리스 버전인 Gazebo Sim 간의 노드 스폰 방식 차이를 분석하는 데 활용하였습니다. 특히 알고리즘 연산 로그는 완벽히 동작하나 바퀴가 구르지 않던 통신 먹통 현상이 발생했을 때, 최신 Gazebo Sim의 `TwistStamped` 규격 중계 문제를 진단하고 명시적인 명령어 기반 통신 브릿지(`ros_gz_bridge`) 레이어를 가동하는 해결책을 도출하는 데 적극적으로 활용하였습니다.

## 6. 상세 트러블 슈팅 (Trouble Shooting)

### 6.1 시뮬레이터 버전 격차에 따른 모델 스폰 및 이름 변경 불가 문제
* **문제 상황**: 프로젝트 제안서 작성 당시 가정했던 구형 환경과 달리, 실습 환경이 최신 Gazebo Sim 버전으로 구성되어 있었습니다. 이로 인해 기존 표준 서비스 명령어인 `/spawn_entity`를 이용한 임시 모델 생성이 거부되었으며, 마우스 우클릭을 통한 엔티티의 이름 변경(`Rename`) 기능 역시 시뮬레이션 상태에 종속되어 차단되는 문제가 발생했습니다.
* **원인 분석**: 최신 Gazebo Sim은 물리 엔진과 GUI 모델 관리 방식이 전면 개편되어 예전 방식의 서비스 통신 요청을 무시하며, 월드가 일시 정지(Pause)되어 있거나 개체 속성이 고정된 경우 GUI 패널에서의 수동 수정을 차단하기 때문이었습니다.
* **해결 방법**: 특정 객체 이름을 문자열로 일일이 매칭하여 추적하는 소프트웨어 의존성을 완전히 제거했습니다. 대신 가상 시간축의 실시간 흐름을 확보한 뒤, LiDAR 센서 고유의 절대 거리 배열 값 중 최솟값을 실시간 필터링하여 타겟을 잡는 정밀 거리 필터링 알고리즘으로 코드를 개편하여 한계를 근본적으로 해결했습니다.

### 6.2 알고리즘 연산 정상 작동 중 로봇 바퀴 구동 먹통 문제
* **문제 상황**: 주행 노드를 가동했을 때 터미널 창에는 정밀한 거리 및 각도 연산 로그가 실시간으로 완벽하게 출력되었으나, 가상 세계 안의 터틀봇은 바퀴가 전혀 구르지 않고 멈춰 있는 통신 단절 현상이 발생했습니다.
* **원인 분석**: 코드 노드는 연산을 마치고 표준 `/cmd_vel` 토픽(`geometry_msgs/msg/Twist`) 규격으로 바퀴 명령을 전송하고 있었으나, 최신 Gazebo Sim은 내부적으로 헤더 데이터가 포함된 `TwistStamped` 포맷으로 중계를 요구하고 있어 중간 통신망에서 명령이 유실되었던 것이 원인이었습니다.
* **해결 방법**: 터미널 백그라운드에 꼬여있던 유령 프로세스들을 `killall` 명령어로 전면 초기화하여 유동 랙을 제거했습니다. 이후 ROS2와 Gazebo Sim 사이의 통신 규격을 중계해 주는 매개체인 `ros_gz_bridge parameter_bridge /cmd_vel@geometry_msgs/msg/Twist@gz.msgs.Twist` 명령어를 독립 터미널 창에 가동하여 주행 신호를 모터 축에 다이렉트로 주입함으로써 최종 추종 주행을 성공시켰습니다.
